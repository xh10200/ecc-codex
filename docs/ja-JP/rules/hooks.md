# ローカル自動化と検証

このルールは Codex ワークフロー向けのローカル自動化ポイントを説明するもので、
実行可能なグローバル Hook 登録ではありません。自動化はリポジトリ内のスクリプト、
タスクランナー、エディタアクション、git hooks、CI、またはエージェントが明示的に
実行する検証コマンドとして表現してください。

## 推奨フェーズ

- **編集前**: 影響範囲を確認し、対象パッケージやプロジェクトを特定して、最小の検証コマンドを選ぶ
- **編集後**: 変更したファイルに対してフォーマット、lint、型チェック、またはパッケージ単位の検証を行う
- **終了前**: 変更リスクに応じた最小十分の最終チェックを実行する

## 権限と自動化

注意して運用:
- マシン全体の設定ではなく、リポジトリ内コマンドを優先する
- 広範囲で重いチェックより、狭く明確なチェックを優先する
- 危険なグローバル自動化で Codex の承認やサンドボックス制御を回避しない
- 重要で繰り返すチェックはスクリプト、`Makefile`、パッケージマネージャーのコマンドとしてリポジトリに残す

## 計画のベストプラクティス

複数ステップの作業では:
- 短いタスク計画を保つ
- 大きな編集の前に目標を確認する
- 高リスクまたはブロッカーになり得る検証を早めに出す
- 終了前に何を検証したかを明確に残す

良い計画で見つけやすくなるもの:
- 順序が誤っているステップ
- 欠けている検証
- 不要な作業
- 変更範囲の広げすぎ
- 要件の読み違い
